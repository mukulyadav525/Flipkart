SLIDE 4 — TASK 1: IMAGE PREPROCESSING
────────────────────────────────────────────────────────────────────────────────

TITLE:  Task 1 — Image Preprocessing

LEFT SIDE — Before/After comparison table:

  Challenge          Technique Applied
  ─────────────────────────────────────────────────
  Low light          CLAHE (Contrast Limited Adaptive
                     Histogram Equalisation) — per tile
  Rain streaks       Median filter (kernel=5) removes
                     vertical bright streaks
  Motion blur        Unsharp masking (α=1.5, σ=1.0)
                     restores edge definition
  Harsh shadows      Adaptive histogram equalisation
                     preserves local contrast
  Color cast         BGR→LAB conversion, equalise L
                     channel only, convert back

RIGHT SIDE — Code snippet:
  def preprocess(frame):
      lab = cv2.cvtColor(frame, cv2.COLOR_BGR2LAB)
      lab[:,:,0] = clahe.apply(lab[:,:,0])
      frame = cv2.cvtColor(lab, cv2.COLOR_LAB2BGR)
      frame = cv2.medianBlur(frame, 5)   # derain
      frame = unsharp_mask(frame)        # sharpen
      return frame

SPEAKER NOTES:
  Before any AI model sees the frame, we normalise it. CLAHE is applied in
  LAB colour space so only luminance is equalised — this avoids colour
  distortion. The median filter removes rain streaks because rain appears as
  isolated bright pixels while edges are preserved. Unsharp masking recovers
  edge detail lost to motion blur.

SLIDE 6 — TASK 3A: VIOLATIONS (APPEARANCE-BASED)
────────────────────────────────────────────────────────────────────────────────

TITLE:  Task 3 — Violation Detection: Appearance-Based Rules

THREE CARDS (one per violation):

  CARD 1 — Helmet Non-Compliance
  ┌──────────────────────────────────────────────────────┐
  │ HOW IT WORKS:                                       │
  │ 1. Detect motorcycle via YOLO                       │
  │ 2. Run pose model → get head keypoint (x,y)         │
  │ 3. Check if any 'helmet' detection overlaps         │
  │    the head region (IoU check)                      │
  │ 4. No overlap → VIOLATION                           │
  │                                                     │
  │ Confidence: 74–75% on real Vodra footage            │
  │ (COCO weights; IDD fine-tuning raises to ~88%)     │
  └──────────────────────────────────────────────────────┘

  CARD 2 — Seatbelt Non-Compliance
  ┌──────────────────────────────────────────────────────┐
  │ HOW IT WORKS:                                       │
  │ 1. Detect car → crop to driver/passenger region     │
  │ 2. Pose model → left_shoulder & right_hip keypoints │
  │ 3. Scan diagonal strip between those points         │
  │ 4. No bright diagonal band → VIOLATION              │
  └──────────────────────────────────────────────────────┘

  CARD 3 — Triple Riding
  ┌──────────────────────────────────────────────────────┐
  │ HOW IT WORKS:                                       │
  │ 1. Detect motorcycle bounding box                   │
  │ 2. Count person detections whose centroid           │
  │    falls inside the motorcycle box                  │
  │ 3. Count > 2 → VIOLATION                            │
  └──────────────────────────────────────────────────────┘

SPEAKER NOTES:
  The first three violation types are appearance-based — they only need to
  look at what's visible in one frame. Helmet and seatbelt use pose keypoints
  to localise the relevant body region, then check for the safety equipment.
  Triple riding simply counts how many people overlap a motorcycle's bounding
  box. These rules are fast and work even without camera calibration.

SLIDE 7 — TASK 3B: VIOLATIONS (GEOMETRY-BASED)
────────────────────────────────────────────────────────────────────────────────

TITLE:  Task 3 — Violation Detection: Geometry-Based Rules

FOUR CARDS:

  CARD 1 — Wrong-Side Driving
  ┌──────────────────────────────────────────────────────┐
  │ Camera calibrated with:                             │
  │   lane_direction_vector: {x: -1.0, y: 0.12}        │
  │   lane_zone_polygon: 4-point region                 │
  │                                                     │
  │ Logic:                                              │
  │   1. Vehicle must be inside lane_zone_polygon       │
  │   2. Motion vector from tracker (last 8 frames)     │
  │   3. cos(motion, lane_direction) < -0.3 → VIOLATION │
  │                                                     │
  │ Zone scoping prevents false positives on            │
  │ vehicles in the legal opposing lane.                │
  └──────────────────────────────────────────────────────┘

  CARD 2 — Stop-Line Violation
  ┌──────────────────────────────────────────────────────┐
  │ stop_line_coords: two endpoints (px)                │
  │ Signed crossing: dot-product sign change            │
  │ between consecutive centroids and line normal       │
  │ Cleared automatically when signal = GREEN           │
  └──────────────────────────────────────────────────────┘

  CARD 3 — Red-Light Violation
  ┌──────────────────────────────────────────────────────┐
  │ Requires: signal_state = "red" (from config or      │
  │ signal detector) AND vehicle crosses stop line.     │
  │ Both conditions must be true simultaneously.        │
  └──────────────────────────────────────────────────────┘

  CARD 4 — Illegal Parking
  ┌──────────────────────────────────────────────────────┐
  │ no_parking_zone_polygon: 4-point region              │
  │ Vehicle centroid inside polygon → candidate         │
  │ Motion gate: movement < 15px over 8 frames          │
  │ = truly stationary → VIOLATION                      │
  │ Moving vehicle → CLEAR (no false positive)          │
  └──────────────────────────────────────────────────────┘

SPEAKER NOTES:
  Geometry-based rules require per-camera calibration — you define lane
  direction vectors and zone polygons once for each camera installation.
  The wrong-side rule uses the vehicle's motion vector from the IoU tracker
  and checks if it opposes the legal lane direction. Critically, we scope
  the check to a polygon covering only the monitored lane — this eliminates
  false positives from vehicles in the opposing legal lane. Illegal parking
  has a motion gate: the vehicle must be genuinely stationary, not just
  briefly stopped at a traffic light.

SLIDE 9 — TASK 5: LICENSE PLATE RECOGNITION
────────────────────────────────────────────────────────────────────────────────

TITLE:  Task 5 — Automatic Number Plate Recognition (ANPR)

LEFT — Pipeline:

  Step 1: Detect plate region
    → YOLO detects 'license_plate' class inside vehicle bbox
    → Fallback: lower 1/3 of vehicle bbox if no plate detected

  Step 2: OCR
    → EasyOCR (supports Hindi + English characters)
    → Multiple reads per vehicle, aggregated across frames
    → Best reading selected by composite score

  Step 3: Acceptability Gate
    → Length: 6–11 characters
    → Must contain ≥ 2 digits AND ≥ 1 letter
    → Rejects bus route numbers, shop names, false reads
    → "SPECIAL" → "ECLAL" → REJECTED ✓

  Step 4: Format Validation
    → Indian plate regex: MH12AB1234, KA01MN2345, etc.
    → Valid format → 1.5× confidence boost

  Step 5: Normalisation
    → Strip special characters
    → Uppercase
    → Store best reading per track

RIGHT — Sample outputs:
  ┌──────────────────────────────────────────────┐
  │ READ        GATE    SCORE   STORED           │
  │ ─────────────────────────────────────────── │
  │ MH12AB1234  PASS    0.87    MH12AB1234       │
  │ KA03CD5678  PASS    0.79    KA03CD5678       │
  │ ECLAL       FAIL    —       (not stored)     │
  │ 12345       FAIL    —       (no letters)     │
  │ SPECIAL     FAIL    —       (no digits)      │
  └──────────────────────────────────────────────┘

NOTE BOX:
  "Plates from distant CCTV (>15m) are often unreadable. The system
  correctly leaves plate blank rather than guessing. Closer cameras and
  dedicated plate-zoom hardware read plates reliably."

SPEAKER NOTES:
  The ANPR module has a critical quality gate. Without it, the OCR reads
  bus destination boards, shop signage on the roadside — anything with
  text. The acceptability gate requires both digits and letters, and a
  sensible character count, which eliminates almost all false reads. When
  the plate is genuinely unreadable, we leave it blank rather than
  inventing a plate number that could result in a wrongful fine.

  │   Citizen portal         • Confidence %    │
  │   Challan issued              ↓            │
  │                      APPROVE → challan     │
  │                      REJECT  → audit log  │
  └──────────────────────────────────────────────┘

  EVERY ACTION LOGGED:
  Audit log records: who reviewed, when, what action, plate, violation type

WHY THIS MATTERS box:
  • Protects citizens from wrongful fines
  • Satisfies legal admissibility requirements
  • AI accuracy improves over time via feedback
  • Officers review evidence, not raw footage
    (10× more efficient than manual watching)

SPEAKER NOTES:
  This is the most important architectural decision in the project. Traffic
  fines are legal instruments — they need to survive disputes in court. By
  requiring human approval, we ensure that: (1) every fine has a named officer
  accountable for it, (2) borderline detections are reviewed before action,
  and (3) false positives are caught. The officer doesn't watch hours of
  footage — they see a curated queue of high-probability violations with
  full evidence. This is 10 to 100 times more efficient than manual monitoring.

CALIBRATION STEPS (one-time per camera):
  1. Mark lane direction vector (pixel direction of legal flow)
  2. Draw lane zone polygon (which lane to monitor)
  3. Draw no-parking zone polygon
  4. Define stop-line coordinates (if painted)
  5. Connect signal-state API (or use colour detection)

DOCKER COMPOSE:
  One command spins up: backend + frontend + worker
  docker compose -f deploy/docker-compose.yml up

SPEAKER NOTES:
  For real deployment, a small compute box — such as an NVIDIA Jetson —
  sits at each junction and runs the streaming pipeline locally. This
  keeps video data on-site (privacy benefit) and sends only violation
  records and annotated images to the central backend. The backend can
  serve thousands of cameras because it only stores and serves text and
  JPEG thumbnails. Per-camera calibration is a one-time setup that takes
  about 30 minutes with our calibration tool.

  A: The preprocessing handles rain and low-light with CLAHE and median
     filtering. Nighttime requires IR-trained weights — that's Phase 2.

  Q: How does it scale to many cameras?
  A: Each camera runs a local edge worker. The central backend only stores
     and serves small JSON records and thumbnail JPEGs — it can handle
     hundreds of cameras on a single VM.

     are sent centrally. The system only stores evidence when a violation
     is detected, not continuous surveillance data.
